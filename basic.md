# 基于分支的二方依赖管理

## 使用依赖列表

`apply from: resources.text.fromInsecureUri("http://www.ossplatform.com/devopsall/dependency/" + depBranch + "/dependency.gradle?pwd="+gradle_dependency_devops_pwd)`

注意这里除了需要提供密码外, 依赖分支 `depBranch` 也需要通过property参数来提供

## 分支

* 新建依赖分支

  所有依赖分支都必须来自一个已经存在的分支,比如202603分支可以来自于202510分支, 这时只需要将202510中所有依赖列表copy到202603分支即可.

  在决定统一发版时会新建分支. 未来进入增量维护阶段当需要开启一个发版窗口时(如新项目或计划迭代)也需要创建一个新分支.
* 分支的变更

  一些操作会导致而分支中某些二方包的版本发生变化,此时依赖管理系统并不是简单更新这个版本, 而是在数据库中新建一行记录,同时记录变更的时间戳, 这可以为依赖管理服务提供依赖信息"闪回"能力.
* 分支的合并

  原则上已经创建的分支不需要合并, 但如果底层竖井处于管理因素决定提供单独分支用于某些开发场景(如SDK改造就是这类), 后续又希望将通过测试的版本直接纳入更新的分支,可以考虑使用合并.因为版本是单调递增的,故合并的动作只是简单将源分支中版本高于目标分支的部分覆盖即可.

## 删除操作

随着时间推移, 部分二方包不再使用时, 不可直接删除, 必须拉新的分支后在新的分支上删除, 删除动作完成前,该分支对流水线不可见.
